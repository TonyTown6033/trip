# 信息收集流程可行性分析

基于现有技术栈（Python/FastAPI、高德地图 API、FlyAI CLI、和风天气 API、MediaCrawler）评估流程图各环节。

## ✅ 可实施

| 步骤 | 工具 | 说明 |
|------|------|------|
| 设计总路线 | Agent + 本地 Markdown | 已有 `02-route/` 结构 |
| 酒店/民宿 | FlyAI CLI | 已配置，可按城市查询 |
| 景点 | 高德地图 API | 已有 key，POI 搜索 |
| 价格调查 | FlyAI CLI | 支持 `--sort price_asc` |
| 设计城市计划 | Agent + `03-stages/` | 结构已就绪 |
| 串联 | Web 前端展示 | FastAPI 已有 |

## ⚠️ 有限制

| 步骤 | 问题 |
|------|------|
| 60天天气 | 和风天气免费版仅7天预报，付费版最多30天。60天只能用**历史同期数据**作参考，不是真正的预报 |
| 小红书爬虫 | MediaCrawler 已迁到 `05-research/tools/MediaCrawler`，CDP 模式可复用已登录 Chrome；仍需人工登录确认和小范围采集 |
| 美食信息 | 没有现成数据源。高德 POI 能搜餐厅但无评价质量，需要小红书或大众点评补充 |
| 用户自定义信息 | 需要定义具体是什么，目前没有对应的采集入口 |

## 结论

大框架可行，两个瓶颈：

1. **天气数据**：建议改为"查询历史同期气候 + 出发前7天滚动更新实时预报"，而非一次性获取60天预报（技术上不存在这种服务）。
2. **口碑/评价数据**（小红书）：MediaCrawler 登录态已验证可通过 CDP 复用本机 Chrome，但仍只能用于小范围公开信息复核；采集结果需作为趋势和线索，再用官方/半官方来源核验事实。
